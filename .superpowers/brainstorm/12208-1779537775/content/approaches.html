Proposed Approaches

Recommended design for each of the 4 features

1. Style Preference Setup

Recommended: DB column (style_preference) + radio picker on signup + editable in settings + wire through cron.

Why this way

  • AI engine already supports all 3 styles
  • One DB migration, two UI changes, one cron wiring change
  • Minimal scope, maximum impact

Not doing

  • Per-client style overrides (v2)
  • Style preview in settings (nice-to-have)

2. UPI Auto-Generation

Recommended: Call createPaymentLink() inside the invoice creation server action. Add "Regenerate Link" button on edit page for invoices with failed/expired links.

Why this way

  • Single-file change to invoices-actions.ts
  • Reuses existing createPaymentLink() and payment-link API
  • Regenerate button handles error recovery

Not doing

  • Auto-regeneration on edit (only manual button)
  • Bulk payment link generation

3. Client Risk Cards

Recommended: Replace table with visual cards in a responsive grid. Add search bar, risk filter (All/Low/Med/High), and sort (name/outstanding/risk).

Why this way

  • Matches PRD spec for visual risk cards
  • Progress bars and risk badges make data scannable
  • Each card clickable → new client detail page

Not doing

  • Table toggle (adds complexity, no clear need)
  • Inline editing on cards

4. Response History (Client Detail Page)

Recommended: New /clients/[id] route with client info header + reminder timeline + outstanding invoices + key stats.

Why this way

  • Clean separation — list page stays scannable, detail page has full context
  • Timeline view shows every follow-up with delivery/response status
  • Creates foundation for future client detail features

Not doing

  • Edit capability on detail page (separate edit page exists)
  • Response logging UI (requires WhatsApp Business API)
